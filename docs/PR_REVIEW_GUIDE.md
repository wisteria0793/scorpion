# PR レビュー対応ガイド

このリポジトリで私（またはレビュワ）が Pull Request をレビューする際に行うこと、期待される対応をまとめます。

目的: 品質・セキュリティ・動作を担保しつつ、早くマージできる状態に整えるためのチェックリストです。

チェックポイント（レビュワが行うこと）

- 自動テストの結果を確認する（CI がある場合は必須）
- ユニットテスト・統合テストをローカルで実行して失敗がないか確認
- Linter/フォーマット (ESLint / Black など) に従っているか
- セキュリティ観点: ハードコードされた秘密情報やログ出力に敏感情報が含まれていないか
- API 契約や DB スキーマ変更があれば、ドキュメント・マイグレーションが適切に追加されているか
- UX / UI 変化がある場合はスクリーンショットや簡易操作手順が PR に含まれているか
- 依存関係の追加は最小限で、バージョンを管理しているか
- 変更が大きい場合は段階的（小さな PR）に分割できないか提案

レビュワが指摘したときに行うべき対応（PR 作成者）

- 指摘に対してコード修正を行い、コミットを追加して再実行する（必要であれば単一コミットに squash して整理）
- 変更点を PR コメントで説明（なぜ修正したか）
- テストが失敗した場合は失敗の原因を明示してから修正
- セキュリティ懸念がある場合はチームに知らせ、別 PR に分割して対応することを検討

マージ方針（小さなプロジェクト向け）

- 少数の変更（単一機能・明確なテストあり）はレビュワの承認後マージ
- 大きな変更は小さい PR に分割して段階的にマージ

私がレビュー対応を「行う」とは具体的に:

1. PR を受け取る（既に作った PR にコメントを書きます）
2. 自分でテスト／lint を走らせる（もし CI が無ければローカルで実行）
3. 必要に応じてコード修正を私が実装し、コミット->PR に push して再レビュー待ちへ持っていく
4. 設定やドキュメント不足があれば補う（README、ドキュメント、テスト）
5. 全てクリアしたらマージまで行います（またはマージ済みのブランチを指示します）

---

疑問点があれば、このファイルか PR のコメント欄で聞いてください。レビューの粒度やチームのルールに合わせて柔軟に調整します。
